Starting a Project from Zero
CLI · folder structure · mental model · 4 scenario types

Step 1 — Generate the project

Open your terminal in the folder where you want the project to live, then run:

# ng new <project-name> creates everything you need
# --routing  → generates app.routes.ts automatically
# --style=css → use plain CSS (or scss, sass, less)
ng new book-app --routing --style=css

# Enter the folder and start the dev server
cd book-app
ng serve

Step 2 — Understand what was generated

The CLI creates a complete project. Here is the part you will actually touch:

src/
  app/← everything Angular lives here
    app.ts← Root component — the shell that wraps every page
    app.html← Root template — contains <nav> and <router-outlet />
    app.config.ts← Providers: router, HTTP client, etc.
    app.routes.ts← All URL → component mappings
    components/← one sub-folder per feature/page (you create these)
      home/
        home.ts   home.html   home.css
    services/← business logic & data fetching (you create these)
    models/← TypeScript interfaces describing your data shapes
  main.ts← Entry point — boots the app (rarely touched)
  index.html← The single HTML page the browser loads (rarely touched)
  styles.css← Global styles (navbar, body, resets)
angular.json← CLI config (build settings, assets)

Step 3 — Ask yourself 5 questions before writing any code

These questions map directly to the Angular building blocks you will need:

1
What pages / URLs does my app have?
Create one component per page. Register each one in app.routes.ts. The component file goes in src/app/components/.
2
Does any page display data from somewhere (API, local list)?
Define the data shape as a TypeScript interface (models/book.ts). Create a service to fetch/manage it (services/book.service.ts). If the data comes from an HTTP API, add provideHttpClient() to app.config.ts.
3
Does any page have a form where the user types something?
Add ReactiveFormsModule to that component's imports. Use FormBuilder to define fields and validation rules.
4
Are any routes private (login required)?
Create an auth service that tracks login state. Create a guard function (auth.guard.ts) and add canActivate: [authGuard] to the protected routes.
5
Do multiple unrelated components need to share the same data?
Put that shared state inside a service (with a signal or observable). Inject the service in both components — they will automatically stay in sync.

Step 4 — Match your project to a scenario type

A
Static / Presentational App
Pages that just show fixed content. No API, no forms.
  • Components (one per page)
  • Routing in app.routes.ts
  • <nav> with routerLink in root template
  • <router-outlet /> in root template
  • That's it. No service, no HTTP.
B
Data-Driven App
Pages that fetch and display data from an API or service.
  • Everything from Scenario A
  • TypeScript interface — the data shape (models/book.ts)
  • Service with HttpClient or local data
  • provideHttpClient() in app.config.ts
  • Component fetches via service in ngOnInit()
  • Template displays with @for, @if, async pipe
C
App with User Input
Scenario B + forms where the user creates or edits data.
  • Everything from Scenario B
  • ReactiveFormsModule imported in the form component
  • FormBuilder to define fields and validators
  • [formGroup] and formControlName in the template
  • Service method to POST or PUT the data
  • Error messages shown conditionally with @if
D
Authenticated App
Scenario C + protected routes that require login.
  • Everything from Scenario C
  • AuthService tracks isLoggedIn state
  • Login component with a form
  • authGuard function redirects if not logged in
  • canActivate: [authGuard] on protected routes
  • HTTP interceptor adds auth token to requests (advanced)

Step 5 — Typical build order

Follow this order to avoid "I don't know where to start" paralysis:

  1. Plan routes first. List every URL and which component shows there. Write them in app.routes.ts even before the components exist.
  2. Generate components. ng g c components/home — one per route. Add them to the routes file.
  3. Build the navigation. Add routerLink links in app.html and verify routing works with placeholder content.
  4. Define your data model. Write the TypeScript interface first — it forces you to think about data shape before writing any logic.
  5. Create the service. ng g s services/book — start with hardcoded data, then swap in HttpClient.
  6. Wire components to the service. Inject the service, fetch data in ngOnInit(), display it in the template.
  7. Add forms. Only after display is working — add create/edit forms with validation.
  8. Add guards last. Protect routes once the app is functional.

How an Angular App Starts
main.ts → app.config.ts → root component

src/main.ts This project

// This is the single entry point — the browser loads this first.
// Its only job is to boot the Angular app.
import { bootstrapApplication } from '@angular/platform-browser';
import { appConfig } from './app/app.config';
import { App } from './app/app';

// bootstrapApplication takes:
//   1. The root component (App) — the top-level shell
//   2. appConfig — where providers (router, http, etc.) are registered
bootstrapApplication(App, appConfig)
  .catch(err => console.error(err));
Modern Angular (v14+) — no AppModule Older tutorials use NgModule + platformBrowserDynamic().bootstrapModule(AppModule). Modern Angular replaces all of that with this single bootstrapApplication() call.

Components — The Building Block
@Component · selector · imports · template · styles

Anatomy of a standalone component This project

// Every component is a TypeScript class decorated with @Component.
// The decorator tells Angular: "this class is a UI component".
import { Component, signal } from '@angular/core';
import { RouterOutlet, RouterLinkWithHref, RouterLinkActive }
  from '@angular/router';

@Component({
  // The CSS selector used to place this component in an HTML file.
  // e.g. <app-root /> in index.html
  selector: 'app-root',

  // IMPORTANT: In standalone components, you declare HERE what
  // other components, directives, and pipes this template uses.
  // No shared NgModule needed — each component is self-contained.
  imports: [RouterOutlet, RouterLinkWithHref, RouterLinkActive],

  // The HTML template for this component's view
  templateUrl: './app.html',

  // CSS that is scoped ONLY to this component — won't leak out
  styleUrl: './app.css'
})
export class App {
  // A signal holding the app title.
  // 'protected' means only this class and its template can read it.
  // 'readonly' means it can't be reassigned (only signal.set() can change the value).
  protected readonly title = signal('book-app');
}

Templates & Data Binding
The 4 ways component class and HTML talk to each other
{{ }} — Interpolation  ·  class → DOM (text only)
// component class name = 'Angular'; title = signal('book-app'); // template (HTML) // Angular replaces {{ }} with the property's string value <h1>Hello, {{ name }}</h1> // Signals must be called like a function: title() <p>App: {{ title() }}</p>
[ ] — Property Binding  ·  class → DOM
// component class isDisabled = true; imgUrl = 'logo.png'; // template — sets a DOM property from class data // The value inside "" is a TypeScript expression, not a string <button [disabled]="isDisabled">OK</button> <img [src]="imgUrl" />
( ) — Event Binding  ·  DOM → class
// component class onSave() { /* called when button is clicked */ } onType(e: Event) { /* called on every keystroke */ } // template — calls a class method when the event fires // $event is the native DOM event object <button (click)="onSave()">Save</button> <input (input)="onType($event)" />
[( )] — Two-Way  ·  class ↔ DOM
// component class // needs FormsModule in @Component imports searchTerm = ''; // template — typing in the input updates searchTerm, // and changing searchTerm in code updates the input <input [(ngModel)]="searchTerm" /> <p>You typed: {{ searchTerm }}</p>
From this project — app.html [routerLinkActiveOptions]="{ exact: true }" is property binding: the value { exact: true } is a JavaScript object passed to the routerLinkActive directive. Square brackets always mean "evaluate the right side as TypeScript, not as a plain string."

Routing
Routes · RouterOutlet · routerLink · params · navigate

app.routes.ts This project

// Routes is an array of objects.
// Each object maps a URL path → a component to show.
export const routes: Routes = [

  // Empty path '' = the homepage (URL is just '/')
  { path: '', component: Home },

  // Static path — shows BookList at /books
  { path: 'books', component: BookList },

  // Dynamic segment: :bookId is a parameter.
  // /books/42 → BookDetails gets bookId = '42'
  { path: 'books/:bookId', component: BookDetails },

  // Redirect: old path → new path
  // pathMatch:'full' means only redirect exact empty string, not sub-paths
  { path: '', redirectTo: 'home', pathMatch: 'full' },

  // Wildcard ** catches everything that didn't match above.
  // MUST be last — Angular checks routes in order, top to bottom.
  { path: '**', component: NotFound }
];

Reading route parameters — when a URL has :id

// When the route is: { path: 'books/:bookId', component: BookDetails }
// and the URL is /books/42, we need to extract that '42' inside the component.
import { ActivatedRoute } from '@angular/router';
import { inject } from '@angular/core';

export class BookDetails implements OnInit {
  // ActivatedRoute gives us access to the current route's parameters
  private route = inject(ActivatedRoute);

  ngOnInit() {
    // Option 1 — snapshot: reads the value once, right now.
    // Use this when the parameter won't change while the component is open.
    const id = this.route.snapshot.paramMap.get('bookId');

    // Option 2 — observable: updates reactively if the URL param changes.
    // Use this when the user can navigate from /books/1 to /books/2
    // without leaving the BookDetails component.
    this.route.paramMap.subscribe(params => {
      const id = params.get('bookId');
    });
  }
}

Programmatic navigation (from TypeScript)

// Sometimes you need to navigate after an action,
// not from a click on an <a> tag.
private router = inject(Router);

goToBook(id: number) {

05
Dependency Injection & Services
@Injectable · inject() · providedIn · singleton pattern
What is Dependency Injection? Instead of a component creating its own service with new BookService(), Angular creates the service once and gives it to any component that asks for it. This means all components share the same instance — so state is automatically shared.

Creating a service

// A service is a plain class with @Injectable.
// It holds logic and state that multiple components can share.
import { Injectable } from '@angular/core';

// providedIn: 'root' means Angular creates ONE instance
// for the whole app (singleton). Available everywhere.
@Injectable({ providedIn: 'root' })
export class BookService {

  private books: Book[] = [
    { id: 1, title: 'Clean Code' }
  ];

  getBooks() {
    return this.books;
  }
}

Injecting — two styles

// ── Modern style (Angular 14+) — recommended ──────
// inject() can be used directly in field declarations
export class BookList {
  private svc = inject(BookService);
  books = this.svc.getBooks();
}

// ── Classic style — constructor injection ─────────
// Angular sees the type annotation and injects it
export class BookList {
  constructor(private svc: BookService) {}

  ngOnInit() {
    this.books = this.svc.getBooks();
  }
}
Prefer inject() in modern Angular It works in field initialisers, is easier to test, and keeps the constructor clean.

Providing at different levels

Signals — Modern Reactive State
signal() · computed() · effect() — Angular 17+
What is a Signal? A signal is a box that holds a value. Angular watches that box. When the value changes, Angular automatically re-renders any template or computed() that reads it. No manual subscriptions, no detectChanges().

Already in this project — app.ts This project

// signal('book-app') creates a signal holding the string 'book-app'
protected readonly title = signal('book-app');

// In the template, read a signal by CALLING it like a function:
// {{ title() }}  — note the parentheses!

Full signals API with examples

import { signal, computed, effect } from '@angular/core';

export class CounterComponent {

  // signal(initialValue) — writable signal
  // Reading: this.count()    Writing: this.count.set(5)
  count = signal(0);

  // computed() — a derived value that recalculates automatically
  // when any signal it reads changes. Read-only — no .set()
  doubled = computed(() => this.count() * 2);

  constructor() {
    // effect() runs a side effect whenever a read signal changes.
    // Angular automatically tracks which signals are read inside.
    effect(() => {
      console.log('count changed to:', this.count());
    });
  }

  increment() {
    // update() receives the previous value and returns the new one
    this.count.update(prev => prev + 1);
  }

  reset() {
    // set() replaces the value directly
    this.count.set(0);
  }
}

Signal methods

MethodWhat it does
signal(val)Create a writable signal
sig()Read the current value
sig.set(val)Replace value directly
sig.update(fn)Update based on previous value
computed(fn)Read-only derived signal
effect(fn)Side effect on signal change
Signals vs Observables Signals are synchronous, always have a value, no subscribe needed. Perfect for component state and simple reactive data.

Observables are async streams — many values over time. Use them for HTTP calls, real-time data, and complex async flows.
07
Lifecycle Hooks
Angular calls these methods automatically at specific moments
constructor()
Class is created. @Input values not set yet. No DOM.
→ Inject services here only. No logic.
ngOnChanges(changes)
Called every time an @Input() / input() value changes from the parent. Also fires once before ngOnInit.
→ React to parent data changes.
ngOnInit()
Called once after the first ngOnChanges. Inputs are now available. The most commonly used hook.
→ Fetch data, subscribe to observables, initialise state.
ngAfterViewInit()
Called after the component's HTML and all child components are rendered into the DOM.
→ Access @ViewChild elements, interact with the DOM directly.
ngOnDestroy()
Called just before Angular removes the component from the DOM.
→ Unsubscribe from observables, clear timers. Critical for preventing memory leaks.

Typical pattern with HTTP fetch

import { OnInit, OnDestroy }
  from '@angular/core';

// Declare which hooks you implement
export class BookDetails
  implements OnInit, OnDestroy {

  private svc = inject(BookService);
  book: Book | null = null;

  // Store the subscription so we can cancel it
  private sub!: Subscription;

  ngOnInit() {
    // Good place to fetch data because
    // inputs (like route params) are already set
    this.sub = this.svc
      .getBook(this.id)
      .subscribe(book => {
        this.book = book;
      });
  }

  ngOnDestroy() {
    // Prevents memory leaks when the component
    // is removed (e.g. user navigates away)
    this.sub.unsubscribe();
  }
}
08
Input & Output — Component Communication
signal inputs · signal outputs · parent↔child data flow
The mental model Data flows DOWN from parent to child via inputs (property binding [prop]). Events flow UP from child to parent via outputs (event binding (event)). Components don't reach into each other — they communicate through this contract only.

New syntax — Angular 17+  Recommended

Instead of decorators, inputs and outputs are now defined as signal functions. Inputs become read-only Signals — you read them with (), just like any other signal.

Child component — book-card.ts
// Import input and output from '@angular/core' // NOT the old Input/Output decorators import { Component, input, output } from '@angular/core'; @Component({ selector: 'app-book-card', ... }) export class BookCard { // input<Type>(default) — optional input // Returns a Signal. Read inside class: this.title() // Read in template: {{ title() }} title = input<string>(''); author = input<string>(''); // input.required<Type>() — parent MUST pass this. // No default needed. Angular errors at compile time if missing. bookId = input.required<number>(); // output<Type>() — replaces EventEmitter entirely // Parent listens with: (selected)="handleIt($event)" selected = output<number>(); onSelect() { // bookId is a Signal — read it with bookId() this.selected.emit(this.bookId()); } }
Child template — book-card.html
<!-- Inputs are Signals: call them with () --> <div (click)="onSelect()"> <!-- title() reads the signal's current value --> <h3>{{ title() }}</h3> <p>by {{ author() }}</p> </div>
<!-- ─────────────────────────────────────── --> <!-- Parent template — books.html --> <!-- Template syntax is IDENTICAL to the old way. --> <!-- Angular wires the signals under the hood. --> @for (book of books; track book.id) { <app-book-card [title]="book.title" [author]="book.author" [bookId]="book.id" (selected)="handleSelected($event)" /> }

Old syntax — decorator-based  Still works, but verbose

You will see this in older code and many tutorials. Know both.

// Import the decorators — @Input and @Output
import { Component, Input, Output, EventEmitter }
  from '@angular/core';

export class BookCard {

  // @Input() makes 'title' settable from the parent.
  // It is a PLAIN TypeScript property — read as this.title (no parentheses)
  @Input() title:  string = '';
  @Input() author: string = '';

  // required: true — Angular warns if the parent forgets to pass it.
  // The ! tells TypeScript "I know this will be set before use".
  @Input({ required: true }) bookId!: number;

  // @Output() + new EventEmitter<T>() creates an event the parent can catch.
  // T is the type of the value emitted — here a number.
  @Output() selected = new EventEmitter<number>();

  onSelect() {
    // bookId is a plain property — read directly, no ()
    this.selected.emit(this.bookId);
  }
}

Required input@Input({ required: true }) bookId!: numberbookId = input.required<number>()

Read in template{{ title }}{{ title() }}

The parent template never changes Whether you use the old or new syntax inside the child, the parent always uses the same [prop]="value" and (event)="handler($event)" syntax. Angular handles the wiring.

Built-in Directives
@if · @for · @switch · *ngIf · *ngFor · ngClass · ngStyle

New control flow Angular 17+

<!-- @if — conditionally show/hide content -->
<!-- The condition is any TypeScript expression -->
@if (books.length > 0) {
  <p>Found {{ books.length }} books</p>
} @else {
  <p>No books found.</p>
}

<!-- @for — loop over an array -->
<!-- track tells Angular which field identifies each item -->
<!-- Without track, Angular re-renders the whole list on any change -->
@for (book of books; track book.id) {
  <app-book-card [title]="book.title" />
} @empty {
  <!-- shown when the array is empty -->
  <p>The list is empty.</p>
}

<!-- @switch — like a JS switch statement in the template -->
@switch (status) {
  @case ('loading') { <p>Loading...</p> }
  @case ('error')   { <p>Something went wrong.</p> }
  @default            { <p>Ready.</p> }
}

Classic directives Still valid

<!-- *ngIf — show element only when condition is true -->
<div *ngIf="isLoggedIn">Welcome!</div>

<!-- *ngIf with else template -->
<div *ngIf="isLoggedIn; else noAccess">
  Welcome!
</div>
<ng-template #noAccess>
  Please log in.
</ng-template>

<!-- *ngFor — loop — trackBy is the performance key -->
<li *ngFor="let book of books;
  trackBy: trackById">
  {{ book.title }}
</li>

<!-- [ngClass] — add/remove CSS classes conditionally -->
<p [ngClass]="{
  'active': isActive,
  'has-error': hasError
}"></p>

<!-- [ngStyle] — apply inline styles dynamically -->
<div [ngStyle]="{ color: textColor }"></div>
Always use track in @for track book.id tells Angular which property uniquely identifies each item. Without it, when the array changes Angular destroys and recreates every DOM element — even if only one item changed. Always track.

Custom pipe

// A pipe is a class decorated with @Pipe.
// 'name' is what you use in templates: | truncate
import { Pipe, PipeTransform }
  from '@angular/core';

@Pipe({
  name: 'truncate',
  standalone: true
})
export class TruncatePipe
  implements PipeTransform {

  // transform(value, ...args) is always the method name.
  // value = whatever is left of the |
  // limit = the argument after : in the template
  transform(
    value: string,
    limit = 50
  ): string {
    return value.length > limit
      ? value.slice(0, limit) + '…'
      : value;
  }
}
<!-- To use: import TruncatePipe in @Component imports -->
<p>{{ book.description | truncate:80 }}</p>

HTTP & Observables
HttpClient · RxJS Observable · async pipe · subscribe

1 — Register the provider in app.config.ts

// Without this line, injecting HttpClient anywhere in the app will crash.
// Add it once and it becomes available everywhere via inject(HttpClient).
import { provideHttpClient } from '@angular/common/http';

providers: [ provideRouter(routes), provideHttpClient() ]

2 — Define the data shape (interface)

// Create src/app/models/book.model.ts
// TypeScript interfaces describe the shape of objects.
// They exist only at compile time — no runtime cost.
export interface Book {
  id:     number;
  title:  string;
  author: string;
}

3 — Create a service with HttpClient

import { HttpClient } from '@angular/common/http';
import { inject, Injectable } from '@angular/core';
import { Observable } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class BookService {

  // HttpClient is Angular's built-in HTTP service
  private http = inject(HttpClient);
  private url  = 'https://api.example.com/books';

  // http.get<T>() returns an Observable<T>.
  // Observables are LAZY — nothing happens until someone subscribes.
  getBooks(): Observable<Book[]> {
    return this.http.get<Book[]>(this.url);
  }

  getBook(id: number): Observable<Book> {
    // Template literals build the URL: /books/42
    return this.http.get<Book>(`${this.url}/${id}`);
  }

  create(b: Book): Observable<Book> {
    // POST sends the book object as the request body
    return this.http.post<Book>(this.url, b);
  }

  update(b: Book): Observable<Book> {
    return this.http.put<Book>(`${this.url}/${b.id}`, b);
  }

  remove(id: number): Observable<void> {
    return this.http.delete<void>(`${this.url}/${id}`);
  }
}

Option A — manual subscribe

// You manage the subscription yourself.
// Advantage: full control.
// Risk: forgetting to unsubscribe causes memory leaks.
export class BookList
  implements OnInit, OnDestroy {

  private svc = inject(BookService);
  books: Book[] = [];
  private sub!: Subscription;

  ngOnInit() {
    this.sub = this.svc
      .getBooks()
      .subscribe({
        next:  data => this.books = data,
        error: err  => console.error(err)
      });
  }

  ngOnDestroy() {
    // ALWAYS unsubscribe in ngOnDestroy
    this.sub.unsubscribe();
  }
}

Option B — async pipe Preferred

// The component just exposes the Observable.
// The template subscribes via the async pipe.

Component class

import { ReactiveFormsModule,

  from '@angular/forms';

  imports: [ReactiveFormsModule],
  ...

export class LoginForm {

  // FormBuilder is a helper that creates form controls
  private fb = inject(FormBuilder);

  // fb.group() creates a FormGroup (the whole form).
  // Each key is a field name. The array is [initialValue, validators].
  form = this.fb.group({
    email: [
      '',
      // Validators run on every change. All must pass for field to be valid.

      '',
      [Validators.required, Validators.minLength(6)]
    ]
  });

    // form.valid is true only if ALL fields pass their validators
    if (this.form.valid) {
      // form.value is a typed object: { email: string, password: string }
      console.log(this.form.value);
    }
  }
}

Route Guards
Run logic before entering or leaving a route

auth.guard.ts

// A guard is a function (not a class) that returns:
// true  → allow navigation
// false → block silently
// UrlTree → redirect to a different route
import { inject } from '@angular/core';
import { CanActivateFn, Router }
  from '@angular/router';

export const authGuard: CanActivateFn = () => {
  // inject() works inside guard functions too
  const auth   = inject(AuthService);
  const router = inject(Router);

  // If logged in: let them through (return true)
  // If not: redirect to /login
  return auth.isLoggedIn()
    ? true
    : router.createUrlTree(['/login']);
};

Apply in app.routes.ts

// Add canActivate to any route that needs protecting
{
  path: 'books',
  component: BookList,
  // Array — you can stack multiple guards
  canActivate: [authGuard]
},
{
  path: 'admin',
  component: Admin,
  // Both guards must return true for navigation to proceed
  canActivate: [authGuard, adminGuard]
}

Signals — Modern Reactivity
signal() · computed() · effect() — Angular 17+
What is a Signal? A reactive value Angular tracks automatically. When a signal changes, any template or computed() that reads it re-renders. Always synchronous, always has a current value.

Already in this project — app.ts This project

protected readonly title = signal('book-app');
// In template: {{ title() }}  ← call it like a function

Full signals API

import { signal, computed, effect } from '@angular/core';

export class CounterComponent {
  count   = signal(0);                              // 1. writable signal
  doubled = computed(() => this.count() * 2);    // 2. derived — recomputes automatically

  constructor() {
    effect(() => console.log('count:', this.count())); // 3. side effect
  }

  increment() { this.count.update(v => v + 1); }  // update from previous value
  reset()     { this.count.set(0); }               // set directly
}

Signal methods at a glance

MethodWhat it does
signal(val)Create a writable signal
sig()Read the current value
sig.set(val)Replace value
sig.update(fn)Update based on previous
computed(fn)Read-only derived signal
effect(fn)Run side effect on change
Signals vs Observables Signals — synchronous, always have a value, no subscription needed. Best for component state.

Observables — async streams, multiple values over time. Use for HTTP, DOM events, timers.

Lifecycle Hooks
ngOnInit · ngOnChanges · ngAfterViewInit · ngOnDestroy
constructor()
Class is created. Inputs not set yet.
→ Inject services only.
ngOnChanges(changes)
Called every time an @Input() value changes. Runs before ngOnInit on first render.
→ React to parent data changes.
ngOnInit()
Called once after first ngOnChanges. Inputs are now available.
→ Fetch data, subscribe to observables.
ngAfterViewInit()
Called after the component's view and children are fully rendered.
→ Access @ViewChild, DOM refs.
ngOnDestroy()
Called just before the component is removed from the DOM.
→ Unsubscribe, clear timers, clean up.

import { HttpClient } from '@angular/common/http';
import { inject, Injectable } from '@angular/core';
import { Observable } from 'rxjs';

export interface Book { id: number; title: string; author: string; }

@Injectable({ providedIn: 'root' })
export class BookService {
  private http = inject(HttpClient);
  private url  = 'https://api.example.com/books';

  getBooks():            Observable<Book[]> { return this.http.get<Book[]>(this.url); }
  getBook(id: number): Observable<Book>   { return this.http.get<Book>(`${this.url}/${id}`); }
  create(b: Book):       Observable<Book>   { return this.http.post<Book>(this.url, b); }
  update(b: Book):       Observable<Book>   { return this.http.put<Book>(`${this.url}/${b.id}`, b); }
  remove(id: number):  Observable<void>   { return this.http.delete<void>(`${this.url}/${id}`); }
}
